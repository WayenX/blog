nginx匹配规则

Jun 25, 2018

虚拟主机匹配规则

当一个名称被多个虚拟主机的server_name匹配成功，优先级规则如下：

  1. 准确匹配server_name
  2. 通配符在开始时匹配server_name成功
  3. 通配符在结尾时匹配server_name成功
  4. 正则表达式匹配server_name成功

在以上匹配中，如果server_name被处于同一优先级的匹配方式多次匹配成功，则首次匹配成功的虚拟主机处理请求

#location块配置
location [ = | ~ | ~* | ^~ ] url {…}
url——待匹配的请求字符串

不含正则表达标准url
含正则表达正则url

  • 不加时，niginx服务器首先在server块的多个location块中搜索是否有标准url和请求的字符串匹配，如果有多个可以匹配，就记录匹配最高的一个。然后，服务器再用location块中的正则url和请求的字符串匹配，当第一个正则url匹配成功，结束搜索，并用这个location块处理此请求；如果正则匹配全部失败，就用刚才记录的匹配度最高的location块处理些请求。
  • “=” 用于标准url前面，要求请求字符串与url严格匹配，如果匹配成功，停止向下搜索并立即处理此请求。
  • “~” 用于表示url包含正则表达式，并且区分大小写。

  • “~*” 用于表示url包含正则表达式，不区分大小写。

    url包含正则表达式必须用~或~*处理
  • “^~” 用于标准url之前，要求Niginx服务器找到标识url和请求字符串匹配度最高的location后，立即使用此location处理请求，而不再使用location块中的正则url和请求字符串做匹配。
  • 以=开头表示精确匹配,如 A 中只匹配根目录结尾的请求，后面不能带任何字符串。
  • ^~ 开头表示uri以某个常规字符串开头，不是正则匹配
  • ~ 开头表示区分大小写的正则匹配;
  • ~* 开头表示不区分大小写的正则匹配
  • / 通用匹配, 如果没有其它匹配,任何请求都会匹配到

顺序 no优先级：
(location =) > (location 完整路径) > (location ^~ 路径) > (location ~,~* 正则顺序) > (location 部分起始路径) > (/)

上面的匹配结果
按照上面的location写法，以下的匹配示例成立：

/ -> config A
精确完全匹配，即使/index.html也匹配不了
/downloads/download.html -> config B
匹配B以后，往下没有任何匹配，采用B
/images/1.gif -> configuration D
匹配到F，往下匹配到D，停止往下
/images/abc/def -> config D
最长匹配到G，往下匹配D，停止往下
你可以看到 任何以/images/开头的都会匹配到D并停止，FG写在这里是没有任何意义的，H是永远轮不到的，这里只是为了说明匹配顺序
/documents/document.html -> config C
匹配到C，往下没有任何匹配，采用C
/documents/1.jpg -> configuration E
匹配到C，往下正则匹配到E
/documents/Abc.jpg -> config CC
最长匹配到C，往下正则顺序匹配到CC，不会往下到E
